📚 Git 开发规范（最佳实践）

规范的核心目的是：保持历史清晰、保证主分支稳定、便于回溯

一、分支管理规范

这是团队协作的基石，通常采用 Git Flow 的简化版。

分支类型对照表

分支类型 命名规则 说明 存活周期
主分支 main / master 生产环境。永远稳定，禁止直接 Push，只能通过合并请求（PR/MR）合并 永久
开发分支 dev / develop 测试环境。日常开发集成的分支，功能测试通过后合并于此 永久
功能分支 feat/xxx 新功能。从 dev 切出，开发完合并回 dev 临时（用完即删）
修复分支 fix/xxx Bug修复。从 dev 或 main 切出 临时
热修复 hotfix/xxx 紧急修复。直接从 main 切出，修完合并回 main 和 dev 临时

命名建议

二、Commit 提交规范

清晰的提交记录能让未来的你感谢现在的自己。业界通用的标准是 Conventional Commits。

提交格式

常用类型（Type）

类型 说明 示例
feat 新功能 feat(cart): add checkout button
fix 修补 Bug fix(auth): resolve login crash
docs 仅文档变更 docs: update API description
style 格式调整（不影响逻辑） style: format code with prettier
refactor 代码重构 refactor: simplify user logic
chore 构建/工具变动 chore: update dependencies

三、工作流红线（绝对禁止）

❌ 禁止直接向 main 或 master 分支 Push 代码
必须走 Pull Request/Merge Request 流程

❌ 禁止提交大文件
如 .zip、.sql、视频等，应使用 Git LFS 或文件服务器

❌ 禁止在公共分支（如 dev）上使用 git push --force
除非你非常清楚后果

❌ 禁止提交本地配置文件
如 .env、config.local.js，必须在 .gitignore 中排除

🛠️ Git 常用命令详解

启动与基础操作

命令 说明 常用参数/示例
git clone 克隆远程仓库 git clone <url>
git add 将文件加入暂存区 git add .（所有）
git add file.js（指定）
git commit 提交到本地仓库 git commit -m "feat: init"
git commit --amend（修改上一次提交）
git status 查看文件状态 检查哪些文件被修改或未暂存
git diff 查看代码差异 git diff（工作区对比暂存区）
git diff --staged（暂存区对比仓库）

分支操作

命令 说明 示例
git branch 查看/创建分支 git branch（查看）
git branch feat-xxx（创建）
git checkout 切换分支 git checkout dev
git checkout -b feat-xxx（创建并切换）
git switch 切换分支（新版推荐） git switch -c feat-xxx
git merge 合并分支 git merge feat-xxx
git branch -d 删除分支 git branch -d feat-xxx（已合并）
git branch -D feat-xxx（强制删除）

同步与远程

命令 说明 示例
git pull 拉取并合并 git pull origin dev
git fetch 仅拉取不合并 git fetch origin
git push 推送代码 git push origin feat-xxx
git push -u origin feat-xxx（首次关联）
git remote 管理远程仓库 git remote -v（查看远程地址）

“后悔药”操作（撤销与回退）

这是新手最容易慌张的地方，请根据场景选择：

场景 你的需求 推荐命令 风险提示
改错了文件 丢弃工作区的修改 git checkout -- <file>
git restore <file>		 ⚠️ 修改彻底丢失
add 错了 撤销暂存（保留修改） git reset HEAD <file>
git restore --staged <file>		 ✅ 安全
commit 错了 撤销提交（保留修改） git reset --soft HEAD^ 修改还在暂存区
commit 错了 撤销提交（保留代码但不暂存） git reset HEAD^ 修改回到工作区
彻底重来 回滚到某次提交 git reset --hard <commit> 🔴 高危，之后代码全丢
误删提交 找回刚才撤销的提交 git reflog 找到ID后 reset 💊 救命稻草

解决冲突

当你执行 pull 或 merge 时，如果提示 CONFLICT，请按以下步骤操作：

  1. 查看状态：git status（查看哪些文件冲突）
  2. 手动修改：打开冲突文件，找到 >>>>>> 标记，保留需要的代码，删除标记
  3. 标记解决：git add <file>
  4. 完成合并：git commit（Git 会自动生成合并提交信息）

🚀 标准开发流程演示

假设你要开发一个”用户登录”功能，标准流程如下：

1️⃣ 同步代码（确保基于最新代码开发）

2️⃣ 创建分支

3️⃣ 开发与提交（小步提交）

4️⃣ 推送分支

5️⃣ 发起合并请求

去 GitLab/GitHub 网页，点击 “New Merge Request”

  • 源分支：feat/user-login
  • 目标分支：dev
  • 指派同事进行 Code Review

6️⃣ 清理（合并成功后）

💡 专家提示

关于 Rebase（变基）

git rebase 可以让提交历史变成一条直线，非常整洁。

  • ✅ 单人开发分支：推荐用 rebase
  • ⚠️ 团队协作合并：推荐用 merge
  • ❌ 公共分支（如 dev）：慎用 rebase，会修改提交历史，可能导致队友的代码冲突

关于 .gitignore

在项目一开始就配置好 .gitignore，排除以下文件：

  • node_modules/
  • *.log
  • .DS_Store
  • .env
  • dist/ 或 build/

这是专业素养的体现。

🚑 误删分支恢复指南

别慌！在 Git 中，删除分支通常只是删除了一个”指针”，真正的代码提交（Commit）往往还在仓库里，只要没有被垃圾回收机制清理掉，就有很大的概率可以恢复。

🔴 立即停手

⏰ 时效性
git reflog 默认只保留 30-90 天的记录（取决于配置），过期后会被自动清理。所以恢复操作越早越好。
