这套 SkyWalking + Prometheus + Grafana 的组合，可以说是目前 Java 微服务架构下的”黄金搭档”。它之所以在生产环境中如此流行，是因为它完美解决了”黑盒”问题——SkyWalking 负责看清调用关系和慢请求，Prometheus 负责监控资源水位和趋势，而 Grafana 则负责把这些数据变成直观的仪表盘。

🩺 第二部分：生产环境问题排查指标指南

当生产环境出现问题时，不要盲目地翻日志。请按照以下场景，查看对应的指标和工具。

场景 1：用户反馈”系统很卡”或”接口超时”

看哪里？ SkyWalking (链路追踪)

核心指标：

  1. 服务拓扑图：看是否有红色的箭头或节点。如果有，说明依赖的服务或数据库有问题。

  2. Endpoint (接口) 响应时间：

    • 进入 SkyWalking 的”服务”页面，查看 Avg SLA (平均响应时间)。
    • 重点关注 P90 / P95 / P99 耗时。如果平均耗时正常，但 P99 很高，说明有少量请求极慢（可能是死锁或 GC 停顿）。

  3. 慢调用链：点击具体的慢接口，查看 Trace 列表，找到耗时最长的那条 Trace。

    • 排查点：看是 SQL 执行慢（数据库瓶颈），还是调用了第三方接口慢，或者是代码逻辑耗时。

场景 2：系统突然报错（500 错误、空指针等）

看哪里？ SkyWalking (错误定位) + ELK/日志 (堆栈详情)

核心指标：

  • SkyWalking 错误率：在服务列表看 Error %。
  • Trace 状态：在”追踪”页面筛选 Status = Error。

排查流程：

  1. 点击红色的 Trace ID。
  2. 找到标红的 Span（出错的那个步骤）。
  3. 查看下方的”标签”或”日志”，通常会直接显示异常信息。
  4. 如果信息不全，复制 TraceId 去日志系统（ELK 或服务器日志文件）搜索，定位具体代码行。

场景 3：服务器 CPU 飙高 或 内存溢出 (OOM)

看哪里？ Grafana (结合 Prometheus)

核心指标：

  1. CPU 使用率：

    • 指标：process_cpu_usage。如果长期接近 100%，说明有死循环或复杂计算。

  2. JVM 内存：

    • 指标：jvm_memory_used_bytes{area="heap"} (堆内存)。
    • 排查点：如果曲线呈锯齿状且峰值越来越高，最后 Crash，说明有内存泄漏。

  3. GC 情况：

    • 指标：jvm_gc_pause_seconds。
    • 排查点：如果 Full GC 频繁（曲线密集），会导致应用”假死”或响应变慢。

场景 4：服务突然不可用（请求打不进去）

看哪里？ Grafana (宏观监控) + Prometheus

核心指标：

  1. QPS (每秒请求数)：

    • 指标：rate(http_server_requests_total[1m])。
    • 排查点：看流量是否突增（被攻击？）或突降（网关挂了？）。

  2. 线程池状态：

    • 指标：tomcat_threads_busy 或 hikaricp_connections_active (数据库连接池)。
    • 排查点：如果线程池满了，说明请求处理不过来，需要扩容或优化代码。

  3. 存活状态：

    • 指标：up{job="spring-microservices"}。如果值为 0，说明 Prometheus 连不上你的服务（服务挂了或网络不通）。

📌 总结：生产环境监控大盘布局建议

在 Grafana 中，建议你为每个核心服务配置一个 Dashboard，包含以下四块核心区域：

  • 系统层：CPU、内存、磁盘 IO、网络流量。
  • JVM 层：堆内存使用、GC 耗时、线程数、类加载数。
  • 应用层：QPS、HTTP 状态码分布（200 vs 500）、接口平均响应时间。
  • 依赖层：数据库连接池状态、Redis 连接数、下游服务调用耗时。
