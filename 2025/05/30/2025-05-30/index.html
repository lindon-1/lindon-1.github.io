🔍 MyBatis-Plus 源码深度解析

导读：MyBatis-Plus (MP) 之所以能成为 Java 开发者的”神器”，是因为它在 MyBatis 原生功能之上，通过”自动装配”和”元数据管理”，把大量的重复劳动（如写简单的 CRUD SQL）自动化了。

要看透 MP 的源码，核心就抓住一条主线：

它是如何把 Java 实体类（Entity）变成 SQL 语句并执行出去的？

我将其核心流程拆解为四个关键阶段：

🚀 一、启动初始化：一切的开始

当你启动 Spring Boot 项目时，MP 并没有闲着，它在后台做了一次**”大扫除”和“大布局”**。

📍 核心入口

MybatisPlusAutoConfiguration

🔧 三大核心任务

1️⃣ 自动装配

Spring Boot 启动时，@EnableAutoConfiguration 会加载 MP 的配置类。

2️⃣ SqlSessionFactory 创建

MP 扩展了原生的 SqlSessionFactoryBean，在初始化时：

3️⃣ Mapper 扫描

通过 MapperScannerConfigurer，MP 扫描所有继承了 BaseMapper 的接口，并将它们注册为 Spring Bean。

💡 关键点

此时，MP 已经准备好接管你的数据库操作了，但它还不知道你的表长什么样，这就需要下一步。

💉 二、SQL 注入：无中生有的魔法

你继承了 BaseMapper，什么都没写就能调用 selectById()。这是因为 MP 在启动时，通过 SQL 注入器，把这些通用方法的 SQL 动态”写”进了 MyBatis 的内存中。

🎯 核心组件

ISqlInjector (默认实现：DefaultSqlInjector)

💻 关键代码逻辑（简化版）

🗣️ 人话解释

就像你入职（启动项目），HR（SqlInjector）直接把你的工位、电脑、权限（MappedStatement）都配好了，不用你自己去申请（写 XML）。

🧬 三、元数据管理：TableInfo 与 TableFieldInfo

MP 之所以能生成 SQL，是因为它通过反射，把你 Java 类里的字段和数据库的列对应起来了。这些映射关系被缓存在 TableInfoHelper 中。

📦 核心类

类名 作用
TableInfo 表级别元数据：表名、主键、实体类类型等
TableFieldInfo 字段级别元数据：字段名、属性名、注解信息等

📋 TableInfo（表信息）

它记录了：

  • ✅ 表名
  • ✅ 主键列名
  • ✅ 主键属性名
  • ✅ 实体类类型

它就像一个**”地图”**，告诉 MP：User 类对应 user 表，主键是 id。

📝 TableFieldInfo（字段信息）

它记录了：

  • ✅ 字段名
  • ✅ 属性名
  • ✅ 类型
  • ✅ 是否逻辑删除
  • ✅ 是否乐观锁
  • ✅ 处理注解（如 @TableField(exist = false)）

💻 关键代码逻辑（生成 SQL 片段）

🗣️ 人话解释

MP 启动时把你的实体类**”拍了一张 X 光片”**（TableInfo），以后写 SQL 直接照着片子填空，不用每次都去反射，性能极高。

🏭 工厂运作流程

📚 核心代码类速查

阶段 核心类 作用
入口 MybatisPlusAutoConfiguration 自动装配配置类
注入 DefaultSqlInjector SQL 注入器
元数据 TableInfoHelper, TableInfo 表与字段元数据管理
执行 SimpleQueryExecutor MyBatis 原生执行器
插件 PaginationInterceptor 分页插件
构造器 QueryWrapper, LambdaQueryWrapper SQL 条件构造

🎯 核心要点回顾

1️⃣ 启动初始化

  • 自动装配 + Mapper 扫描 + SqlSessionFactory 配置

2️⃣ SQL 注入

  • 启动时动态生成 MappedStatement
  • 无需手写 XML，即可使用 BaseMapper 所有方法

3️⃣ 元数据管理

  • TableInfo 缓存实体与表的映射关系
  • 避免重复反射，提升性能

4️⃣ 执行与拦截

  • Wrapper 构建 SQL 条件
  • Interceptor 链实现分页、乐观锁等功能
