🔍 MyBatis-Plus 源码深度解析

MyBatis-Plus (MP) 之所以能成为 Java 开发者的”神器”，是因为它在 MyBatis 原生功能之上，通过”自动装配“和”元数据管理“，把大量的重复劳动（如写简单的 CRUD SQL）自动化了。

要看透 MP 的源码，核心就抓住一条主线：

它是如何把 Java 实体类（Entity）变成 SQL 语句并执行出去的？

我将其核心流程拆解为四个关键阶段：

🚀 一、启动初始化：一切的开始

当你启动 Spring Boot 项目时，MP 并没有闲着，它在后台做了一次”大扫除“和”大布局“。

1.1 核心入口：MybatisPlusAutoConfiguration

🔧 自动装配

Spring Boot 启动时，@EnableAutoConfiguration 会加载 MP 的配置类。

🏗️ SqlSessionFactory 创建

MP 扩展了原生的 SqlSessionFactoryBean，在初始化时：

  • ✅ 配置了数据源
  • ✅ 注册了 MP 特有的插件（如分页插件、乐观锁插件）

📡 Mapper 扫描

通过 MapperScannerConfigurer，MP 扫描所有继承了 BaseMapper 的接口，并将它们注册为 Spring Bean。

关键点： 此时，MP 已经准备好接管你的数据库操作了，但它还不知道你的表长什么样，这就需要下一步。

💉 二、SQL 注入：无中生有的魔法

你继承了 BaseMapper，什么都没写就能调用 selectById()。这是因为 MP 在启动时，通过 SQL 注入器，把这些通用方法的 SQL 动态”写”进了 MyBatis 的内存中。

2.1 核心组件：ISqlInjector

默认实现： DefaultSqlInjector

2.2 源码逻辑解析

步骤 1：检查 MappedStatement

MyBatis 执行 SQL 前，会去 Configuration 中找 MappedStatement（MS 对象）。如果找不到，就会报错。

步骤 2：动态注入

MP 在启动时，会遍历 DefaultSqlInjector 中注册的”方法提供者”（如 InsertMethod、SelectByIdMethod）。

步骤 3：生成 MS

它会根据你的实体类信息，动态构建 MappedStatement 对象，并放入 MyBatis 的配置中心。

2.3 关键代码逻辑（简化版）

💡 人话解释

就像你入职（启动项目），HR（SqlInjector）直接把你的工位、电脑、权限（MappedStatement）都配好了，不用你自己去申请（写 XML）。

🧬 三、元数据管理：TableInfo 与 TableFieldInfo

MP 之所以能生成 SQL，是因为它通过反射，把你 Java 类里的字段和数据库的列对应起来了。这些映射关系被缓存在 TableInfoHelper 中。

3.1 核心类

类名 作用
TableInfo 表信息：记录表名、主键列名、主键属性名、实体类类型等
TableFieldInfo 字段信息：记录字段名、属性名、类型、是否逻辑删除、是否乐观锁等

3.2 TableInfo（表信息）

它就像一个”地图“，告诉 MP：

  • User 类对应 user 表
  • 主键是 id

3.4 关键代码逻辑（生成 SQL 片段）

💡 人话解释

MP 启动时把你的实体类”拍了一张 X 光片“（TableInfo），以后写 SQL 直接照着片子填空，不用每次都去反射，性能极高。

⚙️ 四、执行与拦截：Wrapper 与 插件

当你在代码中调用 userMapper.selectList(queryWrapper) 时，真正的魔法发生了。

4.1 条件构造器 (Wrapper)

QueryWrapper 或 LambdaQueryWrapper 本质上是一个 SQL 构建器。

工作原理

步骤 说明
1. 收集条件 内部维护了一个 ArrayList，当你调用 .eq("name", "Tom") 时，它只是往列表里加了一个”等于”的条件对象
2. 最终转换 在执行前，MP 会遍历这个列表，拼接成 SQL 片段（WHERE name = ?），并收集参数值

4.2 拦截器链 (Interceptor Chain)

这是 MP 最强大的地方。MyBatis 原生支持拦截器，MP 利用这一点实现了分页、乐观锁、多租户等功能。

核心流程

4.3 典型场景源码解析：分页插件

工作流程

步骤 操作
1. 拦截 PaginationInterceptor 实现了 MyBatis 的 Interceptor 接口，拦截 Executor.query 方法
2. 改写 SQL 利用 JSqlParser 工具，将原始 SQL 改写为计数 SQL
3. 执行计数 先执行 SELECT COUNT(*) FROM user 获取总数
4. 添加分页 再改写原 SQL 加上 LIMIT 语句
5. 执行查询 最终发给数据库的是改写后的 SQL

关键代码逻辑（分页插件）

💡 人话解释

拦截器就像是一个”安检员“。你的 SQL 想通过去数据库，必须先经过安检员。安检员一看：”哦，你要分页？那我帮你把 LIMIT 加上，顺便帮你数数总共有多少人。”

🎨 五、其他核心功能解析

5.1 乐观锁插件

原理： 通过拦截 update 操作，自动在 SQL 中添加版本号条件

5.2 逻辑删除

原理： 不真正删除数据，而是更新删除标记字段

5.3 自动填充

原理： 在插入或更新时，自动填充指定字段

📊 六、总结：一张图看懂 MP 源码

如果把 MyBatis-Plus 比作一家代工厂：

🏭 工厂运作流程

阶段 类比 实际工作
启动阶段 工厂装修 自动装配，招聘工人（Mapper 扫描）
准备阶段 制定标准 技术部根据设计图（Entity），画出标准作业程序（TableInfo），并把通用工序（BaseMapper 方法）写成标准文档（SQL 注入）
生产阶段 接单生产 业务员下单（调用 selectList），附带特殊要求（Wrapper），车间主任（Interceptor）检查订单并修改加工指令，最后交给数据库（仓库）取货

🎯 七、学习建议

✅ 如何深入理解 MP 源码

  1. 从 BaseMapper 入手

    • 查看 BaseMapper 接口定义的所有方法
    • 追踪这些方法是如何被注入到 MyBatis 中的

  2. 调试 TableInfo 生成过程

    • 在 TableInfoHelper.initTableInfo() 处打断点
    • 观察实体类是如何被解析成表信息的

  3. 研究拦截器链

    • 阅读 PaginationInterceptor 源码
    • 理解 SQL 是如何被改写的

  4. 实践出真知

    • 尝试自定义一个 MP 插件
    • 实现自己的 SQL 注入器
