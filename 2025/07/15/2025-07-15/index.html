📝 MySQL 三大日志深度解析

在 MySQL 中，binlog、undo log 和 redo log 是三种核心日志，它们协同工作，共同保障了数据库的数据一致性、持久性、可恢复性和高可用性。

简单来说，它们各司其职：

日志 职责 比喻
redo log 保证已提交的事务不丢失 InnoDB 引擎的”崩溃恢复”基石
undo log 保证事务可以回滚，支持快照读 事务的”后悔药”
binlog 记录所有数据变更 主从复制和数据恢复的”档案库”

下面我们来逐一详解。

📊 一、三大日志核心对比

为了让你快速建立整体认知，我们先通过一个表格来对比它们的核心区别：

特性 redo log undo log binlog
所属层级 存储引擎层 (InnoDB) 存储引擎层 (InnoDB) MySQL Server 层
日志类型 物理日志 逻辑日志 逻辑日志
记录内容 “在某个数据页上做了什么修改” “事务执行前的数据状态” “执行的 SQL 语句或行变更事件”
核心作用 保证事务的持久性，实现崩溃恢复 保证事务的原子性，实现事务回滚和 MVCC 实现主从复制、数据恢复和审计

🔄 二、Redo Log：崩溃恢复的保障

redo log 是 InnoDB 存储引擎特有的日志，它的核心使命是保证事务的持久性（Durability）。

2.1 核心原理

MySQL 为了性能，对数据的修改不会直接写入磁盘（随机 I/O 太慢），而是先在内存的**缓冲池（Buffer Pool）**中进行，修改后的数据页被称为”脏页“。

如果此时数据库发生崩溃，内存中的脏页就会丢失，导致数据不一致。redo log 解决了这个问题。

WAL 技术（Write-Ahead Logging）

“先写日志，再写磁盘”

当事务修改数据时，会先将”在某个数据页上做了什么修改“这个操作记录到 redo log 中。因为 redo log 是顺序写入磁盘的，速度非常快。

即使数据库在脏页刷盘前崩溃，重启后 InnoDB 也能通过**重放（Replay）**redo log 中的记录，将数据恢复到崩溃前的状态。

2.2 关键特性

📌 循环写入

redo log 的文件大小是固定的，写满后会覆盖旧的内容，像一个环形缓冲区。

⚙️ 刷盘策略

由参数 innodb_flush_log_at_trx_commit 控制：

值 行为 安全性 性能
1 (默认) 每次事务提交都立即将 redo log 刷到磁盘 ⭐⭐⭐ 最安全 较慢
0 每秒刷盘一次 ⭐ 可能丢失 1 秒内的事务数据 最快
2 每次事务提交只写到系统缓存，由操作系统决定何时刷盘 ⭐⭐ 折中方案 较快

2.3 应用场景

数据库崩溃恢复 - 这是 redo log 最主要的作用，确保在 MySQL 意外宕机重启后，所有已提交的事务数据都能被恢复。

↩️ 三、Undo Log：事务回滚与 MVCC 的基石

undo log 同样是 InnoDB 引擎的日志，它记录了数据被修改之前的旧值，核心作用是保证事务的原子性（Atomicity）。

3.1 核心原理

你可以把 undo log 理解为事务的”后悔药“。当一个事务需要修改一行数据时，InnoDB 会先把这行数据的旧版本信息记录到 undo log 中。

功能 1：事务回滚

如果事务执行过程中出错或收到 ROLLBACK 指令，InnoDB 就会利用 undo log 中记录的旧值，将数据恢复到事务开始前的状态。

功能 2：MVCC（多版本并发控制）

这是 undo log 的另一个重要作用。当一个事务读取数据时，如果这行数据正被其他事务修改，InnoDB 可以通过 undo log 中的历史版本链，找到这行数据在当前事务开始时的版本，从而实现”快照读“，避免了读写冲突，极大提升了并发性能。

3.2 应用场景

场景 说明
事务回滚 当业务逻辑需要撤销一个未完成的事务时，依赖 undo log 来恢复数据
实现 MVCC 在 READ COMMITTED 或 REPEATABLE READ 隔离级别下，普通的 SELECT 查询（快照读）就是通过 undo log 构建的数据历史版本来实现的

📜 四、Binlog：数据同步与恢复的档案

binlog (Binary Log) 是 MySQL Server 层实现的日志，它记录了所有对数据库结构或数据造成变更的操作（如 INSERT, UPDATE, DELETE）。

4.1 核心原理

binlog 是一种追加写入的逻辑日志，它记录了”发生了什么“。

三种记录格式

格式 说明 优点 缺点
STATEMENT 记录执行的原始 SQL 语句 日志量小 某些函数（如 NOW()）可能导致主从数据不一致
ROW 记录每一行数据变更前后的具体值 精确复制，保证数据一致性 日志量大
MIXED 混合模式，MySQL 会根据情况自动选择 兼顾性能和一致性 复杂度较高

4.2 关键机制：两阶段提交 (2PC)

为了保证 redo log 和 binlog 的数据一致性，InnoDB 在提交事务时采用了”两阶段提交“。

  1. Prepare 阶段：InnoDB 将 redo log 写入磁盘，并标记为 prepare 状态。
  2. Commit 阶段：
    • 首先，将 binlog 写入磁盘。
    • 然后，将 redo log 的状态从 prepare 改为 commit。

只有当这两个步骤都成功，事务才算真正提交。这样即使数据库在提交过程中崩溃，重启后也能根据 redo log 和 binlog 的状态判断事务是否应该提交，从而保证主库和从库（依赖 binlog）的数据一致性。

4.3 应用场景

场景 说明
主从复制 主库将数据变更写入 binlog，从库读取并重放这些日志，从而实现数据同步。这是 MySQL 高可用和读写分离架构的基础
数据恢复 当发生误删数据等人为事故时，可以通过解析 binlog，反向生成 SQL 语句，将数据恢复到指定时间点
数据审计 binlog 记录了所有的数据变更历史，可以用于追溯数据变化的来龙去脉

🔐 五、二阶段提交（2PC）深度解析

**二阶段提交（Two-Phase Commit, 2PC）**是 MySQL 中一个至关重要的机制，它的核心使命是保证 redo log 和 binlog 之间的数据一致性。

简单来说，redo log 由 InnoDB 引擎层负责，binlog 由 MySQL Server 层负责。如果没有 2PC，这两个日志在事务提交时就可能“各自为政”，导致数据不一致。2PC 就像一个“协调员”，确保这两个日志要么都成功记录，要么都失败，从而保证数据库在任何情况下（尤其是崩溃后）的数据一致性。

5.1 为什么需要二阶段提交？

我们可以通过一个反例来理解它的重要性。假设没有 2PC，MySQL 先写 redo log，再写 binlog。

场景：一个事务成功写入了 redo log，但在写入 binlog 之前，数据库突然崩溃。

后果：

  1. 重启后：InnoDB 引擎在恢复时，发现 redo log 中有已提交的事务，于是将这个事务的数据恢复到数据页中。此时，主库的数据已经更新。
  2. 主从复制：由于 binlog 中没有这条记录，从库在同步时就会缺失这个事务。

最终问题：主库和从库的数据永久性不一致。

反之，如果先写 binlog 再写 redo log，也会遇到类似问题，最终导致主从数据不一致。

为了解决这个问题，MySQL 引入了 2PC，将事务的提交过程拆分为两个阶段，强制这两个日志的提交具有原子性。

5.2 二阶段提交的具体流程

当一个事务准备提交时，会经历以下两个阶段：

阶段一：准备阶段 (Prepare Phase)

  1. InnoDB 引擎将事务对数据的修改记录到 redo log 中。
  2. 将 redo log 持久化到磁盘，并将该事务的状态标记为 PREPARE。
  3. InnoDB 通知 MySQL Server 层：“我已经准备好了”。

此时，事务处于”半提交“状态，持有锁，等待 Server 层的最终指令。

阶段二：提交阶段 (Commit Phase)

  1. MySQL Server 层收到准备成功的消息后，将事务的操作写入 binlog 并持久化到磁盘。
  2. binlog 写入成功后，Server 层通知 InnoDB 引擎：“可以正式提交了”。
  3. InnoDB 将 redo log 中该事务的状态从 PREPARE 修改为 COMMIT，并释放事务持有的锁等资源。

关键点：只有当 redo log (prepare) 和 binlog (commit) 都成功持久化后，整个事务才算真正完成。

5.3 崩溃恢复：2PC 的价值体现

2PC 的真正价值在数据库崩溃重启时才能完全体现。重启后，InnoDB 会扫描 redo log，并根据不同的状态采取相应的恢复策略，以确保与 binlog 的一致性。

崩溃发生时机 redo log 状态 binlog 状态 恢复动作 原理
Prepare 阶段之后 PREPARE 不存在 回滚该事务 redo log 已准备但 binlog 未写入，说明事务未完整提交，必须回滚以保证一致性
Commit 阶段之后 PREPARE 存在 提交该事务 binlog 已写入，说明 Server 层已确认提交，InnoDB 需要补上最后一步，将事务提交
Commit 阶段之后 COMMIT 任意 直接提交 事务已完整提交，无需额外操作

这个恢复过程的核心思想是：以 binlog 作为事务提交的最终依据。通过检查 redo log 中处于 PREPARE 状态的事务是否在 binlog 中存在，来决定是提交还是回滚，从而完美解决了主从数据一致性的问题。

⚙️ 六、关键配置：实现“双 1”保障

为了最大限度地发挥 2PC 的作用，保证数据的强一致性，生产环境中通常会将以下两个参数设置为 1，这被称为”双 1 配置“。

6.1 innodb_flush_log_at_trx_commit = 1

  • 作用：每次事务提交时，都将 redo log 立即刷到磁盘。
  • 目的：保证 redo log 的持久性，即使系统崩溃，PREPARE 状态的日志也不会丢失。

6.2 sync_binlog = 1

  • 作用：每次事务提交时，都将 binlog 立即刷到磁盘。
  • 目的：保证 binlog 的持久性，确保崩溃恢复时能准确判断事务是否已提交。

只有当这两个参数都为 1 时，才能真正实现 Crash-Safe 的主从一致性。
